Highest Priority Findings

1. OAuth token and mailbox data stored in client-side Flask session in legacy web_app.py
- Severity: High
- File/lines:
  web_app.py:1200 stores access_token in session.
  web_app.py:1235 stores full emails list in session.
  web_app.py:1163-1169 uses implicit OAuth response_type=token and response_mode=fragment.
- Why it matters:
  Flask's default session is a signed client-side cookie. The bearer token and email payloads can be sent to the browser, may exceed cookie limits, and are inappropriate for sensitive mailbox data.
- Required fix:
  Replace client-side session token/email storage with server-side state keyed by a secure session ID.
  Replace implicit flow with OAuth Authorization Code + PKCE and state validation.
  Ensure tokens are never placed in URLs, browser history, logs, or cookies.

  web_app.py:1196 reads token from request args.
  phishing_app.py:356-364 requests response_type=token.

  outlook_scanner.py:41-48 requests response_type=token.

- Required fix:

  Delete or clearly deprecate the implicit-flow code paths.
  Validate state on every callback.
  Add token expiry handling and clear tokens on 401.

3. Main Flask app variants store all auth/mailbox state in module globals
- Severity: High
- File/lines:
  final/app.py:46 scan_results global.
  final/app.py:985-990 _graph_token, _graph_user, _live_emails, _junk_emails, _pending_oauth globals.
  brandnewgui/app.py:409-415 same pattern.
  webdash/app.py:409-415 same pattern.
  brandnewgui/devmode/app.py:708-712 same pattern.
- Why it matters:
  Any second browser/client talking to the same backend shares the active Outlook token, user identity, emails, junk emails, and scan results.
- Required fix:
  Introduce server-side per-session storage.
  Generate a secure session ID for the browser/Electron frontend.
  Store graph token, user info, live emails, junk emails, pending OAuth state, and scan results under that session ID.
  Ensure OAuth callback associates state with the correct session.
  Add cleanup/expiry for inactive sessions and pending OAuth requests.

4. Mutating localhost API routes have no CSRF/origin protection
- Severity: High
- File/lines:
  final/app.py:1101 /api/auth/connect POST
  final/app.py:1246 /api/auth/disconnect POST
  final/app.py:1277 /api/auth/refresh POST
  final/app.py:1434 /api/emails/<idx>/move-to-junk POST
  final/app.py:1456 /api/emails/<idx>/delete POST
  final/app.py:1499 /api/scan/<idx> POST
  final/app.py:1574 /api/scan-all POST
  brandnewgui/app.py and webdash/app.py have equivalent routes.
- Why it matters:
  A malicious local webpage can POST to localhost and trigger connect/disconnect/move/delete/scan actions against the active shared mailbox state.
- Required fix:
  Add CSRF token issuance and verification for every mutating route.
  Add Origin/Referer validation allowing only the app origin, e.g. http://127.0.0.1:<actual-port> and the Electron app origin if applicable.
  Reject mutating requests without valid token/origin.
  Prefer SameSite=Strict cookies if cookies are used.

5. Junk-folder operations can target the wrong inbox email
- Severity: High
- File/lines:
  final/static/js/app.js:2033-2040 replaces state.emails with junk emails.
  final/static/js/app.js:2053 still calls /api/emails/<idx>/move-to-junk.
  final/static/js/app.js:2070 still calls /api/emails/<idx>/delete.
  final/app.py:1309-1313 _get_email_list returns inbox when live.
  final/app.py:1460-1469 delete resolves idx against _get_email_list/live inbox.
  final/app.py:1501-1508 scan resolves idx against _get_email_list/live inbox.
  Same pattern exists in brandnewgui/static/js/app.js:1275-1312 and webdash/static/js/app.js:1124-1161.
- Why it matters:
  When viewing Junk, idx 0 in the UI may refer to junk email 0, but backend delete/scan acts on inbox email 0.
- Required fix:
  Replace index-based API contracts with immutable message IDs and folder context.
  Example routes:
    GET /api/emails?folder=inbox
    GET /api/emails?folder=junk
    POST /api/messages/<message_id>/scan
    POST /api/messages/<message_id>/move-to-junk
    DELETE or POST /api/messages/<message_id>/delete
  Store scan results by message ID, not index.
  Ensure the backend validates that the message ID exists in the current session and expected folder before acting.

6. Legacy web renderer is XSS-prone
- Severity: High
- File/lines:
  web_app.py:983 injects config client_id into an input value without escaping.
  web_app.py:1022-1031 injects email ID, sender name, received date, subject, preview directly into HTML attributes/body.
  web_app.py:900-914 uses innerHTML to render selected email fields/body.
  web_app.py:951-1003 in test_page.py has similar innerHTML patterns, but uses static test data, so lower risk there.
- Why it matters:
  Malicious sender names, subjects, previews, or IDs can execute script in the browser.
- Required fix:
  Retire web_app.py if it is not supported.
  If retained, use templates with autoescaping, or html.escape every dynamic field.
  Avoid inline event handlers such as onclick="selectEmail(...)" with untrusted IDs.
  Render body content via textContent or sanitized HTML only.
  Add regression tests with malicious subject/sender strings.

9. Intent LLM service is network-exposed with no auth/rate limit/input cap
- Severity: High
- File/lines:
  intentllm/serve.py:64 /analyze route in vLLM mode.
  intentllm/serve.py:107 binds host="0.0.0.0".
  intentllm/serve.py:139 /analyze route in Flask mode.
  intentllm/serve.py:195 binds host="0.0.0.0".
- Why it matters:
  If deployed as written, anyone who can reach the port can burn GPU/CPU, extract service behavior, and send large prompts.
- Required fix:
  Bind to 127.0.0.1 by default.
  Add --host option only for explicit remote deployments.
  Add API authentication for remote serving.
  Add request body size limits and reject oversized email_text.
  Add per-IP or token-based rate limiting.
  Handle invalid JSON safely with get_json(silent=True) equivalent.

10. Rich HTML email preview loads attacker-controlled HTML and remote resources
- Severity: Medium
- File/lines:
  final/templates/index.html:390 iframe previewBodyHtml sandbox="allow-same-origin".
  final/static/js/app.js:811 stores raw bodyHtml as data-html.
  final/static/js/app.js:2191-2202 writes htmlContent into iframe document.
- Why it matters:
  Scripts are sandboxed, but remote img/css/resource loads can still trigger tracking beacons and leak that the user opened the email.

  Do not write raw email HTML into the DOM.

Phase 5: Storage, Packaging, and Network Robustness

5. Remove webdash debug=True and network auto-install from build scripts.

Phase 6: Cleanup and Repo Hygiene

Acceptance Criteria
- No supported app uses OAuth implicit grant.
- No access token or mailbox payload is stored in browser cookies or URLs.
- Two simultaneous clients cannot share or overwrite each other's Outlook session.
- Mutating routes reject missing/invalid CSRF token and invalid Origin.
- Message scan/move/delete operations use immutable message IDs and work correctly in inbox and junk folders.
- Email content cannot execute scripts or load remote resources by default.
- Pickle model load fails closed if artifact hash mismatches.
- intentllm/serve.py is not remotely exposed without authentication and rate limits.
- Packaged Electron writes mutable data to a user-data directory.
- External HTTP requests have timeouts.
- Debug mode is not enabled by default in any shipped Flask entrypoint.
